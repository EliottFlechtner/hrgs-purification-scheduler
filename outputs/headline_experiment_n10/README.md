# Headline Experiment: N=10 Optimizer vs. Paper's Fig. 4 Schedule

Per `docs/WbW Plan.md`, Weeks 3-4: run the optimizer at the paper's exact
network configuration (`N=10`, `Ōäō=2 km`, branching `(16,14,1)`, `k=18`
arms, `e_d=0.01`) and compare against their Fig. 4 (`flexible_paper`)
schedule with a **resource-normalized comparison** ŌĆö fix `C(╬Ż)` equal to
the paper's own resource cost and show the found schedule does better at
that fixed cost, not just in some unconstrained absolute sense.

Generated by `beam_search` (Tier 3) and `brute_force_search`
(paper baseline), objective = `maximize_rate_with_fidelity_floor(f_min=0.9)`.

## Reading the result the way the WbW plan asks for

The plan's requirement is: **fix `C(╬Ż)` equal to the paper's baseline
cost, and show the optimizer's schedule does better at that fixed cost.**
Two versions of that comparison are included here, both favorable:

1. **Exact-cost match (100 vs. 100).** Constraining the optimizer to
   spend *exactly* the same 100 units of resource (5 half-RGS copies ├Ś
   2 sides ├Ś N=10 hops, matching the paper's own fairness convention)
   still finds a schedule with a higher rate (4158.14 vs. 4055.92,
   **+2.5%**) by trading a bit of surplus fidelity (0.9168 vs. 0.9295,
   both comfortably above the `f_min=0.9` floor) for higher success
   probability.

2. **Budget-constrained (Ōēż 100).** Allowing the optimizer to spend *up
   to* the paper's cost but not forcing it to spend all of it, it
   chooses to spend only **half** (cost=50) and still clears the
   fidelity floor, because spending more on uniform purification only
   suppresses success probability without being needed once `F Ōēź 0.9`.
   This is the single most citable number: **same feasibility
   guarantee, ~65% higher rate, at half the resource cost.**

Since the optimizer's own unconstrained global optimum (`e_max=200`)
lands at the exact same schedule as the budgetŌēż100 case, this also shows
the paper's cost=100 was never resource-*constraining* for this metric ŌĆö
the true optimum uses less, not more.

## Why the optimizer's schedule wins (structural comparison)

- **Paper's `flexible_paper`** uses a fixed, uniform 5-copy pumping
  structure applied the same way regardless of where fidelity is
  actually being lost ŌĆö it clears the floor with room to spare
  (`F=0.93` vs. `f_min=0.9`), which is wasted budget: every added
  purification round both costs resources and multiplies success
  probability down.
- **The optimizer's schedule** (`beam.span.((hop0.n2.YY+(hop1+hop2))+
  (hop3.n3.ZX_YY+...+hop9.n3.ZX_YY)))`) allocates purification
  non-uniformly: the shallow 3-hop left branch needs almost none (one
  `YY` round near the root), while the deep 7-hop right branch ŌĆö where
  composing many hops erodes fidelity fastest ŌĆö gets a two-round
  `ZXŌåÆYY` purification per hop. It spends exactly enough to clear
  `f_min=0.9` and no more.
- Net effect: comparable or better fidelity margin, roughly double the
  success probability, at half the resource cost ŌĆö because the DAG
  structure is *shaped by where composition actually loses fidelity*
  rather than replaying one fixed circuit uniformly across all hops.

## Reproducing

```bash
cd /home/shark/Documents/hrgs-purification-scheduler
source .venv/bin/activate
PYTHONPATH=src python3 - <<'EOF'
from hrgs_scheduler.cost_functions import ObjectiveConfig
from hrgs_scheduler.models.network_config import NetworkConfig
from hrgs_scheduler.search import beam_search
from hrgs_scheduler.search.brute_force import brute_force_search

net = NetworkConfig.integrating_paper_config(e_d=0.01)
obj = ObjectiveConfig.maximize_rate_with_fidelity_floor(f_min=0.9)

paper = next(r for r in brute_force_search(net, obj, e_max=200) if r.label == "flexible_paper")
optimizer = beam_search(net, obj, e_max=100, beam_width=25)[0]  # resource-normalized (budget <= paper's cost)
EOF
```
